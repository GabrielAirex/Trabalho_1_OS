1. Introdução

Este trabalho explora a programação paralela aplicada à multiplicação de matrizes, um problema computacionalmente intenso com complexidade O(n³) na versão clássica. A multiplicação de matrizes é um exemplo ideal de paralelização porque cada elemento cij da matriz resultado pode ser calculado independentemente dos demais — característica conhecida como Independência dos Dados.


Foram implementados quatro programas em C++:

auxiliar

Gera duas matrizes aleatórias com dimensões fornecidas via argumentos e salva em arquivos.

sequencial

Multiplica as matrizes de forma convencional, com três laços aninhados (baseline).

paralelo_threads

Distribui N/T linhas da matriz resultado entre T threads POSIX (pthreads).

paralelo_processos

Cria P processos filho via fork(), cada um calculando N/P linhas.

2.2 Estratégia de Paralelização

Threads: As matrizes são lidas antes da criação das threads. Todas as threads compartilham a memória das matrizes M1 e M2 (somente leitura) e escrevem em regiões distintas da matriz resultado C, eliminando condições de corrida sem necessidade de mutex. Cada thread salva sua parcela em um arquivo próprio com seu tempo individual.
Processos: As matrizes são lidas pelo processo pai antes do fork(). Os processos filho herdam os dados via copy-on-write, calculam sua faixa de linhas, salvam o resultado em seu arquivo exclusivo e encerram com exit(0). O pai aguarda todos com waitpid(). O tempo total é o maior tempo entre os filhos.

2.3 Distribuição das Linhas

Para T threads (ou P processos), a divisão de N linhas é feita como:

linhas_por_worker = N / T          // divisão inteira
resto            = N % T          // sobra distribuída às primeiras workers
worker[i].inicio = soma das faixas anteriores
worker[i].fim    = inicio + linhas_por_worker + (i < resto ? 1 : 0)

Isso garante que a carga seja balanceada mesmo quando N não é divisível por T.

5. Análise de Speedup e Eficiência

O speedup mede quantas vezes a versão paralela é mais rápida que a sequencial. O speedup ideal (linear) seria igual ao número de workers. Na prática, overhead de criação, sincronização e escalonamento reduz esse valor.

Eficiência = Speedup / N_workers × 100%. Uma eficiência de 100% significaria que cada worker contribui perfeitamente. Na prática, valores acima de 60% são considerados bons para problemas de memória intensiva.

6. Etapa 3 – Discussões

6.1 Por que os resultados observados? (Questão a)

Sim, houve diferença significativa entre as três implementações. Abaixo analisamos cada caso:

Sequencial vs. Paralelo — visão geral
Em todos os tamanhos testados, as implementações paralelas superaram a sequencial. Para a matriz 800×800, threads com T=16 alcançaram speedup de ≈6,2× e processos com P=16 speedup de ≈6,1×. O paralelismo é mais benéfico conforme o problema cresce, pois o custo de criação de workers é amortizado pelo volume de cálculo.
Matrizes pequenas (100×100)
Para matrizes pequenas, o overhead de criar e sincronizar threads/processos é comparável ao próprio tempo de cálculo (~312 µs sequencial). Ainda assim, o paralelismo trouxe ganho graças ao copy-on-write e ao compartilhamento de memória eficiente do sistema operacional. Matrizes muito pequenas em sistemas mono-core poderiam apresentar regressão.
Threads vs. Processos
Threads consistentemente superam processos em matrizes médias (200×200, 400×400) porque compartilham o espaço de endereçamento — sem necessidade de copiar dados. Processos têm overhead de fork() e de I/O separado de cada filho. Entretanto, na matriz 800×800 com P=16, os processos equipararam-se às threads, provavelmente porque o overhead de fork() foi amortizado pelo longo tempo de cálculo, e o isolamento de memória favoreceu o cache L1/L2 por processo.
Plateau em T=8 → T=16 para Threads
Na 800×800, o ganho de T=8 para T=16 em threads foi menor do que de T=4 para T=8. Isso ilustra a Lei de Amdahl: mesmo em problemas altamente paralelizáveis, existe uma porção serial (leitura de arquivo, criação de threads, escrita do resultado) que limita o speedup máximo. Além disso, o número de núcleos físicos da máquina limita o paralelismo real — threads extras disputam os mesmos núcleos por time-slicing.

6.2 Valor ideal de T e P? (Questão b)

Threads — T ideal

Para matrizes maiores (≥ 400×400): T = 8 apresenta a melhor relação speedup/eficiência. T=16 traz ganho marginal adicional com eficiência significativamente menor (~44%), indicando contenção por núcleos de CPU. Recomendação: T = número de núcleos lógicos da máquina.

Processos — P ideal

Para matrizes maiores (≥ 400×400): P = 8 a P = 16. O overhead de fork() é maior que threads, por isso em matrizes médias P=8 é mais eficiente. Para matrizes muito grandes, P=16 equipara-se às threads. Recomendação: P = número de núcleos físicos da máquina.

Conclusão geral sobre T e P: O valor ótimo depende do hardware. Em uma máquina com N núcleos, usar mais de N workers causa context switching excessivo sem ganho real. Para este experimento, com base nos dados coletados, T=8 para threads e P=8 para processos representam o melhor ponto de equilíbrio entre speedup e eficiência.

6.3 Observações Adicionais

  • Lei de Amdahl: O speedup é limitado pela fração serial do programa (leitura de arquivo, alocação de memória, escrita de resultados). Mesmo com 16 workers, o speedup máximo observado foi ≈6×, longe do ideal 16×.
  • Localidade de cache: Cada thread/processo trabalha em linhas contíguas da matriz resultado, o que favorece o acesso sequencial à memória e reduz cache misses.
  • Variabilidade: Processos apresentaram maior desvio padrão que threads em matrizes pequenas, reflexo do overhead variável do fork() e do escalonamento de processos pelo SO.
  • Copy-on-write: O fork() no Linux usa copy-on-write, evitando copiar fisicamente as matrizes M1 e M2 para cada filho — apenas as páginas escritas (resultado) são duplicadas. Isso mitiga parte do overhead de processos.

7. Conclusão

Este trabalho demonstrou na prática os conceitos de programação paralela utilizando threads POSIX e processos Unix para acelerar a multiplicação de matrizes. Os experimentos confirmaram os fundamentos teóricos estudados em aula:


  • A paralelização traz ganhos reais de desempenho, especialmente para problemas de maior dimensão, onde o custo de criação dos workers é amortizado.
  • Threads são mais eficientes que processos para computação intensiva com dados compartilhados, pois evitam overhead de cópia de memória.
  • O ganho nunca é linear (Lei de Amdahl), pois sempre há uma porção serial irredutível.
  • O valor ótimo de T e P está relacionado ao número de núcleos de CPU disponíveis.
  • Para problemas suficientemente grandes e com boa paralelização de dados, como a multiplicação de matrizes, a programação paralela é altamente recomendada.
Resultado principal: Para a maior matriz testada (800×800), a versão paralela com 16 threads alcançou speedup de 6,2×, reduzindo o tempo de 354ms para apenas 57ms — uma redução de 84% no tempo de execução.
